home_controller.dart 
class HomeController extends VeloraController {
  final posts = <Post>[].obs;

  @override
  void onInit() {
    super.onInit();
    run(() => _loadPosts());
  }

  Future<void> _loadPosts() async {
    final res = await
      Velora.api.get<List<Post>>('/posts',
        parser: (d) => (d as List)
            .map(Post.fromJson).toList());
    posts.value = res.data ?? [];
  }
}

// Gate a button by server permission
Can(
  permission: 'posts.manage',
  child: FloatingActionButton(
    onPressed: controller.create,
  ),

— What's included

Auth & Sessions

Laravel Sanctum token flow with mock login for local development. No backend required to start building your UI.

Repository Pattern

Data sources implement one interface. Replace MockDataSource with RemoteDataSource — controllers never change.

Permissions & RBAC

Can and RoleOnly widgets gate UI by server-assigned permission or role. State mirrors the authenticated user.

Notification Infrastructure

FCM/APNs wiring, in-app notification list, unread badge counts, and CLI commands for platform setup.

Form Validation

VeloraFormController accepts 422 responses from Laravel and binds field errors inline — no manual mapping.

CLI Scaffolding

Generate modules, auth flows, and notification setup in one command. Consistent structure across every feature.

GetxService owns shared state. Controllers own UI state.

The hard line between framework layer and app layer is intentional. Services hold cross-screen business data; controllers hold what a single screen needs. Read the architecture guide before building your first module.
