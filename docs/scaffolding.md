# 7 — Scaffold a Module

**What you'll do:** Use the Velora CLI to generate a complete CRUD module — service, repository, data source, controller, and pages — then understand what was generated and how to extend it.

---

## Start a new app

Create a brand new Velora-powered Flutter app:

```sh
dart run velora_cli new my_app
```

Writes a Flutter app skeleton with constructor-injected splash/auth/dashboard modules, a `MaterialApp` wired to a hand-written `AppRouter.onGenerateRoute` switch (Navigator-based, not GetX declarative routing), a theme file, and `.ai/` context files for AI-assisted development.

## Scaffold auth

Add the login screen and auth flow:

```sh
dart run velora_cli make:auth --sanctum
```

Generates `lib/app/modules/auth/` with a login page and an auth controller.

## Scaffold a CRUD module

Generate a complete feature module for any resource:

```sh
dart run velora_cli make:module users --crud
```

Writes `lib/app/modules/users/` containing:

```
users/
  users_module.dart                    ← factory: wires dataSource → repository → service → controller
  users_routes.dart                    ← route name constants for this module
  models/
    users_model.dart
  data/
    users_remote_data_source.dart      ← calls Velora.api
    users_repository.dart              ← implements VeloraRepository<UsersModel, int>
  services/
    users_service.dart                 ← plain class, not GetxService
  controllers/
    users_controller.dart              ← extends VeloraController
  pages/
    users_index_page.dart
    users_show_page.dart               ← --crud only
    users_create_page.dart             ← --crud only
    users_edit_page.dart               ← --crud only
```

There is no `bindings/` directory and no `Bindings` subclass: `${ClassName}Module` is a plain class with a static `controller()` factory method that constructs the data source, repository, service, and controller by hand with constructor injection — the same pattern `velora new` uses for its generated modules. `data/` is flat (no `sources/`/`repositories/` split) and there is no generated mock data source; swap implementations by changing what `${ClassName}Module.controller()` constructs. Services are plain classes, not `GetxService`.

`users_routes.dart` only declares this module's route name constants (plus `show`/`create`/`edit` helpers for `--crud`) — it is **not** wired into a global route table for you. The app generated by `velora new` uses a hand-written `switch` in `AppRouter.onGenerateRoute` (see [Architecture](architecture.md)), so add the new routes and pages there yourself.

## Run a health check

```sh
dart run velora_cli doctor
```

Read-only checks against the current directory: `pubspec.yaml` exists and depends on `velora`; `lib/main.dart` exists and calls `Velora.boot(...)`; any installable plugin package (see `velora install`, below) that's already a pubspec dependency but isn't actually wired into `Velora.boot(plugins: [...])`; and the installed Dart/Flutter toolchain versions. A missing/misconfigured `velora` dependency is a hard failure (nonzero exit); everything else — missing `main.dart`, a missing `Velora.boot()` call, an unwired plugin — is reported as a warning and still exits `0`.

## Install a plugin package

```sh
dart run velora_cli install velora_offline
```

Adds the package to `pubspec.yaml`, wires its plugin into `Velora.boot(plugins: [...])` in `lib/main.dart`, and runs `pub get` — the Laravel-style "install and it's wired for you" step. Pass `--no-wire` to skip editing `main.dart` or `--no-pub-get` to skip fetching dependencies.

## All CLI commands

| Command | What it does |
|---|---|
| `velora_cli new <name>` | Create a new app skeleton |
| `velora_cli make:auth --sanctum` | Scaffold login + auth flow |
| `velora_cli make:module <name> --crud` | Generate a full CRUD module |
| `velora_cli make:notifications` | Generate notification center module |
| `velora_cli install:push --fcm` | Add FCM push setup files |
| `velora_cli install:push --local` | Add local notifications setup files |
| `velora_cli install <package>` | Install a Velora plugin package: add the dependency, wire it into `Velora.boot()`, and run `pub get` |
| `velora_cli doctor` | Check the `velora` dependency, `Velora.boot()` wiring, unwired installed plugins, and toolchain versions |

See [CLI Commands →](commands.md) for the full reference including package-level `flutter analyze` and `flutter test` commands.
